Executive Summary

62%

Overall Readiness

The Immaculate monorepo is a highly ambitious, well-structured neuroscience orchestration harness. The core engine (persistence, event sourcing, checkpoint recovery, Zod validation) and the backend server are production-grade in architecture. However, several hard gaps prevent shipping: zero authentication, a benchmark that blocks the server event loop, broken CI W&B bootstrap (Windows-only Python path in a Linux runner), 14 orphaned API endpoints with no UI surface, and a monolithic 1 059-line client component with a critical Three.js memory leak.

Top 5 Risks

[CRITICAL] No authentication on any endpoint. Any network-accessible client can POST /api/control to pause, reset, or boost the live engine. Zero barriers.
[CRITICAL] Benchmark run blocks the event loop. POST /api/benchmarks/run calls runPublishedBenchmark() synchronously (320–640 ticks with await inside a loop), freezing all WebSocket emissions and REST responses for minutes.
[HIGH] Three.js memory leak in ConnectomeScene. EdgeLink instantiates new THREE.Line(...) inside JSX on every render, and the primitive element never triggers Three.js disposal on remove.
[HIGH] WebSocket reconnection absent in both Dashboard and TUI. If the harness restarts or the connection drops, both UIs go permanently dark with no backoff/reconnect logic.
[MEDIUM] W&B Python bootstrap is Windows-only; CI runs Linux. LOCAL_VENV_PYTHON resolves to Scripts/python.exe. The GitHub Actions job runs ubuntu-latest, so W&B will never be available in CI.

GREEN

Gap & Risk Analysis (Prioritised)

P0 — CRITICAL No authentication on any API endpoint

Every REST endpoint and the WebSocket control stream is fully open. POST /api/control with {"action":"reset"} resets the live engine from any network-reachable client. Minimum fix: shared bearer token via env var + middleware.

Affects: server.ts (all routes)

P0 — CRITICAL Benchmark run blocks the server event loop

runPublishedBenchmark() runs 320–640 ticks in an async loop inside the Fastify route handler. All WebSocket emissions and REST responses are blocked for the duration (up to 30 s on the latency-soak pack). Fix: run in a worker thread (node:worker_threads) or separate child process.

Affects: server.ts:634, benchmark.ts

P0 — CRITICAL Three.js memory leak in ConnectomeScene

EdgeLink mounts new THREE.Line(new THREE.BufferGeometry(), new THREE.LineBasicMaterial(...)) as a <primitive> in JSX. React Three Fiber's primitive does not call .dispose() when the component unmounts or when edges change. Every re-render leaks geometry + material into the GPU. Fix: use <line> declarative primitive or call geometry.dispose() / material.dispose() in a useEffect cleanup.

Affects: connectome-scene.tsx:37-52

P1 — HIGH No WebSocket reconnection in Dashboard or TUI

Both clients open a WebSocket once and never reconnect. A harness restart or 30-second network glitch permanently silences both UIs. Fix: implement exponential backoff reconnect with a max-retries cap.

Affects: dashboard-client.tsx:251-297, tui/src/index.tsx:193-242

P1 — HIGH Path traversal on NWB/BIDS scan endpoints

POST /api/ingest/nwb/scan and /api/ingest/bids/scan accept a raw filePath/rootPath from the request body and pass it directly to path.resolve(). An operator can read any file on the server. Fix: validate that the resolved path starts within an allowed root directory.

Affects: server.ts:380-406, server.ts:478-506, bids.ts:135, nwb.ts:252

P1 — HIGH W&B Python path is Windows-only; CI runs Linux

LOCAL_VENV_PYTHON = …/.tools/wandb-venv/Scripts/python.exe. The GitHub Actions job runs ubuntu-latest. The fallback ("python") will fail unless wandb is pre-installed in the runner's system Python. The wandb:bootstrap script is PowerShell — it can't run on Linux. The CI step npm run benchmark:gate:all will fail if any benchmark tries to publish to W&B.

Affects: wandb.ts:39, .github/workflows/benchmark-gate.yml, scripts/bootstrap-wandb.ps1

P2 — MEDIUM useEffectEvent is an experimental React API

Used 4 times in dashboard-client.tsx (handleSnapshot, runBenchmark, runCognition, publishBenchmarkToWandb, startLatestReplay). This hook is not yet in stable React and its signature may change. Consider replacing with useCallback + explicit dependency management or React Query mutations.

Affects: dashboard-client.tsx:122, 300, 326, 358, 386

P2 — MEDIUM 14 orphaned backend endpoints with no UI surface

Endpoints for checkpoints, integrity details, event log, replay pagination, dataset detail, neuro session detail, active live sources, Ollama model listing, Ollama registration, and topology summary all exist but are never called. This is dead surface area that must be maintained but delivers no user value. Either wire them to a UI panel or document them as operator-only curl APIs.

P2 — MEDIUM Monolithic 1 059-line DashboardClient component

All 10 dashboard panels live in a single component. This creates a massive re-render surface (any state change re-renders everything), makes testing impossible, and violates separation of concerns. Each panel should be its own component with focused state. React.memo or layout-level code splitting would reduce paint costs.

Affects: dashboard-client.tsx (entire file)

P2 — MEDIUM NDJSON event log grows unbounded

Every event is appended to events.ndjson. At 180 ms ticks this adds ~5–10 events/second. After a week of continuous operation the file can exceed 100 MB. No rotation or compaction is implemented. Fix: add log rotation by size or age, or compact events into the checkpoint file and truncate the log after checkpoint.

Affects: persistence.ts:503-533

P3 — LOW hashValue (FNV-1a) duplicated 4 times

Identical implementation in bids.ts, nwb.ts, live-neuro.ts, and neuro-replay.ts. Should be a shared utility in packages/core or a local utils.ts in harness/src.

P3 — LOW No "Stop Replay" control in Dashboard or TUI

Operators can start an NWB replay from both UIs but cannot stop it. The POST /api/neuro/replays/:id/stop endpoint exists and is correct — it just needs a button wired to it.

P3 — LOW Orphaned files in project root

tmp-probe.h5, tui.md.txt, tui_2_readme.md.txt, desktop.ini, and multiple .runtime-smoke-* directories in the project root are not in .gitignore and should not be tracked.

P3 — LOW No lint toolchain configured despite declared script

Root package.json declares "lint": "npm run lint --workspaces --if-present" but no workspace has a lint script or ESLint config. The CI does not run lint. Add ESLint + @typescript-eslint to enforce code quality.
